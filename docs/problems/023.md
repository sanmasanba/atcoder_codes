# 023

## 問題リンク

[ABC267 E - Erasing Vertices 2](https://atcoder.jp/contests/abc267/tasks/abc267_e)

## キーワード

削除順の最大コストを抑えるなら、現在のコスト最小から貪欲に消す

## 何に着目するか

ある頂点を消すコストは「まだ残っている隣接頂点の重み和」です。頂点を消すたび、影響を受けるのはその隣だけで、残る頂点のコストは増えません。

コストが最小の頂点を先に消しても、その選択で最大コストが悪化することはありません。最小値を繰り返し取り出すため、優先度付きキューがそのまま使えます。

## 解法方針

最初に各頂点 `v` のコスト `cost[v]` を、隣接頂点の重みの和として計算します。`cost[v]` をキーに最小ヒープへ入れ、次を `N` 回繰り返します。

1. 現在コストが最小の未削除頂点 `v` を取り出す。
2. 答え候補を `max(answer, cost[v])` で更新する。
3. `v` を削除し、残っている各隣接頂点 `to` のコストから `A[v]` を引く。
4. 更新後の `cost[to]` をヒープへ追加する。

|削除する頂点|隣接頂点 `to` の変化|
|---|---|
|重み `A[v]` の頂点 `v`|`cost[to] -= A[v]`|

ヒープには、更新前の古い値も残ります。取り出した `(c,v)` が今の `cost[v]` と異なる、または `v` がすでに削除済みなら捨てる「遅延削除」にします。

### なぜ最小から消してよいか

どの時点でも、残った頂点のコストは削除により減るだけです。ある時点で最小コストの `v` を後回しにして別の頂点を消しても、`v` のコストはそれ以下にしかなりません。したがって `v` を今消したときのコストを、後で消すことで下げるために他の頂点を先に選ぶ必要はありません。この交換を繰り返すと、最小コストから消す順序へ変形できます。

## tips

### 実装

隣接リストを作る際、辺 `(u,v)` なら初期値として `cost[u] += A[v]`, `cost[v] += A[u]` を加えます。ヒープの要素は `(コスト, 頂点)` にし、`removed[v]` で削除済みかを管理します。

### よくある誤り

- 答えを頂点重みの和全体で更新する。使うのは「削除時点で残る隣の重み和」です。
- 隣接頂点がすでに削除済みでも減算する。残っていない頂点のコストは更新不要です。
- ヒープの古い値をそのまま採用する。現在の `cost` と照合します。

### 計算量

各辺について高々 1 回、隣接頂点のコストを更新します。ヒープ操作を含めて `O((N+M) log N)`、メモリは `O(N+M)` です。
